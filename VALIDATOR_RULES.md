## 1. Общие правила для Частей 2–9

### 1.1. Маркеры WebFetch Protocol (BLOCKER)
В **начале** Части обязан быть:
```
[WEBFETCH-ДЕКЛАРАЦИЯ] Начинаю поиск документов по теме «...». Все URL2 будут получены только через реальный web_fetch. Подбор из памяти модели запрещён.
```

В **конце** Части обязан быть:
```
[WEBFETCH-ПОДТВЕРЖДЕНИЕ] Все URL2 в данной Части получены через реальный web_fetch. Заголовки страниц совпадают с наименованиями документов в карточках.
```

Regex валидатора (без учёта регистра):
```python
r"\[WEBFETCH-ДЕКЛАРАЦИЯ\]"
r"\[WEBFETCH-ПОДТВЕРЖДЕНИЕ\]"
```

### 1.2. Маркеры поиска (BLOCKER, 1:1 к карточкам)
Перед **каждой** карточкой (каждой строкой `URL1:`) обязан стоять маркер поиска.

Допустимые форматы (regex):
```
>>> ПОИСК
>>> ПОИСК: запрос
>>> SEARCH: query
[ПОИСК]
[ПОИСК #3]
WEB_SEARCH: ...
web_search(...)
Запрос #5:
Продолжаю верификацию через web_fetch
```

Валидатор считает: `card_count = content.count("URL1:")` и `search_count = len(SEARCH_DECLARATION_RE.findall(content))`.
Если `search_count < card_count` → BLOCKER.

**Правило:** на каждую карточку минимум один маркер. Лучше больше, чем меньше.

## 3. Формат карточки документа (Части 2, 4–9)

Каждая карточка обязана содержать **все** поля в указанном порядке. Регистр важен.

```
Вид документа: ...
Полное наименование: ...
Орган принятия: ...
Дата и номер: ...
Структурный элемент: ...
Статус: действует
Актуальность редакции: подтверждена на ДД.ММ.ГГГГ
VERIFIED URL2: ДА
URL1: `http://...`
URL2: `https://...`
Заголовок страницы URL2: ...
Сверка реквизитов: дата/номер/полное название — совпало
Значение для темы: ...
```

### Критические детали:
- **`VERIFIED URL2: ДА`** — это **отдельная строка**. Без префикса. НЕ `Статус верификации: VERIFIED URL2=ДА`.
- **`Актуальность редакции:`** — обязательное поле. Без него карточка невалидна.
- **URL1 и URL2** — в начале строки, без `— ` или других префиксов. URL в inline code: `` `https://...` ``.
- **Заголовок страницы URL2** — фактический заголовок со страницы, не ожидаемый.
- **Сверка реквизитов** — текст без URL.

### Regex для поиска карточек:
```python
r"(?m)(?=^(?:\d+\.\s+)?Вид документа:)"
```
Карточка начинается со строки `Вид документа:` (опционально с нумерацией `1. ` перед ней).

---

### 3.2. URL2 в каждом блоке (BLOCKER)
Каждый блок документа, содержащий `Полное наименование:`, обязан содержать `URL2:` в том же блоке.

Валидатор разбивает текст по `Полное наименование:` и проверяет каждый блок.

### 3.3. Проверка заголовков URL2 при захвате (BLOCKER)
При `capture-part-output` валидатор **реально открывает каждый URL2** и сравнивает `<title>` страницы с полем `Заголовок страницы URL2`.

- **OK** — заголовок совпал → публикация разрешена
- **MISMATCH** — страница открылась, но заголовок ≠ declared → BLOCKER. URL2 ведёт не на тот документ.
- **UNVERIFIED** — 403/timeout → warning, не блокирует

**Правило:** вписывать в `Заголовок страницы URL2` только то, что реально видно на странице. Не угадывать.

### 3.4. Проверка структурных элементов (advisory)
Валидатор ищет текст `Структурный элемент` на странице URL2. Не найдено → warning с именем документа и URL. Не блокирует, но показывает реальные проблемы.

## 4. Часть 3 — БЛОКИ ПОИСКА СФЕР ПРАВА (I–XXXVII)

### 4.1. Канонические заголовки VERSION 18 (BLOCKER)
Заголовки блоков должны **точно** совпадать с каноническими. **Без markdown-звёздочек**, без перефразирования.

Regex, который ищет валидатор:
```python
rf"(?m)^\s*(?P<roman>{roman_pattern})\.\s+(?P<title>.+?)(?:\s+[—-]\s+|\s*$)"
```
Где `roman_pattern` = `I|II|III|...|XXXVII`

**Важно:** звёздочки `**I. Заголовок**` **НЕ мешают** — валидатор их убирает: `title = re.sub(r"^\*+|\*+$", "", title).strip()`

НО: если заголовок не совпадает с каноническим — BLOCKER.

### 3.3. URL2 в применимых блоках (BLOCKER)
Если блок помечен как «найдено» или «не выявлено» (не «не применимо»), он обязан содержать минимум одну строку `URL2:`.

Regex:
```python
r"URL2:"
```

**Правило:** если блок «найдено» — дай карточку с URL2. Если «не применимо» — не надо URL2. Если «не выявлено» — тоже не надо URL2, но лучше не писать «не применимо» для блоков которые объективно применимы.

## 4. Research-log (research-log.jsonl)

### 4.1. Формат записей (BLOCKER)
Каждая запись — JSON-объект в одной строке. Обязательные поля:
```json
{"timestamp": "2026-04-11T20:20:00Z", "tool": "web_search", "query": "...", "result": "..."}
{"timestamp": "2026-04-11T20:20:01Z", "tool": "web_fetch", "url": "https://...", "title": "...", "content": "..."}
```

### 4.2. Проверка подлинности (BLOCKER)
Валидатор требует **минимум 2 дополнительных поля** кроме URL и timestamp.
Запись вида `{"url": "...", "timestamp": "..."}` считается фиктивной.

Для `web_fetch` записей обязательно:
- `url` — URL страницы
- `title` — заголовок страницы
- `content` — краткое содержание

Для `web_search` записей обязательно:
- `query` — поисковый запрос
- `result` — результат

### 4.3. Соответствие URL2 (BLOCKER)
Каждый URL2 в тексте Части обязан быть в research-log как запись `web_fetch` с `title` и `content`.
URL2 ищется по substring: если `url2 in logged_url` или `logged_url in url2` — считается найденным.

## 9. Research-log: дополнительные защиты (BLOCKER)

### 9.1. Defense 1 — реальность URL (BLOCKER)
Валидатор **выборочно (5 случайных URL) открывает URL из research-log** и проверяет HTTP 200.
Если URL возвращает 404 → BLOCKER: «лог заполнен несуществующими URL без реального web_fetch».

**Правило:** в research-log вписывать только URL, которые реально существуют и открылись через web_fetch.

### 9.2. Defense 2 — запрет Python-скриптов (BLOCKER)
Валидатор ищет `.py` файлы в корне проекта (кроме `notary_agent.py`), содержащие:
- `research-log`, `research_log`, `url_fetched`, `write.*research`

Если найдены → BLOCKER: «обнаружены Python-скрипты, которые могут генерировать research-log».

**Правило:** НИКАКИХ `.py` файлов в `tmp/` или рядом. Карточки и research-log — только через реальные tool calls.

### 9.3. Defense 3 — кластеризация timestamp (BLOCKER)
Валидатор проверяет:
- Нужно **минимум 10 записей** в research-log. Если < 10 — проверка пропускается.
- Если span (разница между первым и последним timestamp) **≤ 30 секунд** → BLOCKER: «лог сгенерирован пакетом».
- Если интервалы между записями **идеально равномерны** (max - min < 1.0 сек) → BLOCKER: «признак машинной генерации с постоянным шагом».
- Записи с `"agent_supplied": true` пропускаются этой проверкой.

**Правило:** timestamp в research-log должны быть **неравномерными** (разница между интервалами ≥ 1 сек) и охватывать **минимум 31 секунду** реального поиска. Пример: 19:30:12, 19:31:45, 19:33:02, 19:35:18...
