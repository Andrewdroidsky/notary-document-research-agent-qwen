## Правильные роли: Qwen vs notary_agent.py

### notary_agent.py — оркестратор
Никогда не генерирует substantive-текст самостоятельно. Его функции:
- создаёт workspace и пакеты (prepare-part-XX-plan, run-subtopic)
- валидирует результат (capture-part-output)
- собирает финал и публикует (assemble-subtopic-final)

### Qwen (LLLM) — генератор substantive-текста
Единственный, кто генерирует содержательный текст:
- поиск документов через web_search / web_fetch
- карточки документов с URL1/URL2, сверкой реквизитов, VERIFIED URL2
- аналитика Частей 2, 5, 6–10
- мини-конспект (Часть 10), практические задания (Часть 11)

### Человек
- запускает команды оркестратора
- проверяет результат
- даёт команды на продолжение

### Правильный workflow

**⚠ ОБЯЗАТЕЛЬНЫЙ НУЛЕВОЙ ШАГ перед любым web_search:**

```
python notary_agent.py prepare-part-02-web <subtopic_id>
```

Эта команда создаёт run и пустой research-log. Запрещено начинать web_search до её выполнения. Запрещено создавать run вручную через mkdir. Запрещено писать research-log задним числом.

1. `prepare-part-02-web <subtopic_id>` → run создан, research-log пустой, launch packet готов
2. Qwen читает launch packet — в нём точный путь к research-log
3. Qwen генерирует текст Части через web_search / web_fetch, каждый вызов сразу пишет в research-log
4. `capture-part-output` → агент валидирует и захватывает
5. Повторять п.1–4 для каждой Части (prepare-part-XX-plan для Частей 3–9)
6. `assemble-subtopic-final` → сборка и публикация

### Запрещено
- Писать собственные скрипты конвертации (md_to_docx.py) в обход capture-part-output
- Говорить "передаю управление агенту" — это бессмысленно, агент текст не генерирует
- Утверждать, что агент сам ищет документы или генерирует текст

### Правильная команда для запуска подтемы
`run-subtopic` → получить пакет → Qwen генерирует текст → `capture-part-output` → повторить для каждой Части

## Антигаллюцинационный протокол (07.04.2026) — ОБЯЗАТЕЛЕН ДЛЯ КАЖДОЙ ЧАСТИ 2–9

### Уровень 1 — Пороги URL

| Метрика | Порог | Режим |
|---------|-------|-------|
| URL2 | 35 | жёсткий блокер |
| URL2 advisory | 50 | ориентир |
| URL1 | 45 | только ориентир |

Не гнаться за счётчиком — каждая карточка только через реальный поиск.

### Уровень 2 — Commitment Protocol: маркер поиска перед каждой карточкой

Перед каждой карточкой в тексте Части обязан стоять маркер поиска. Допустимые форматы:
```
>>> ПОИСК
[ПОИСК #N]
WEB_SEARCH: ...
Запрос #N:
Продолжаю верификацию через web_fetch
```
Требование: строго **1:1** — один маркер на одну карточку. `search_count < card_count` → `RuntimeError` при захвате.

### Уровень 3 — Grounded Research: один поиск → одна карточка

Цикл работы — без исключений:
```
web_search → web_fetch → ОДНА карточка → web_search → web_fetch → ОДНА карточка → ...
```
Накапливать результаты поиска и писать карточки блоком — запрещено.

### WebFetch Protocol — три обязательных маркера в теле Части

Все три маркера пишутся **внутри текста Части** (не в чате). Валидатор проверяет их при `capture-part-*`. Отсутствие любого → `RuntimeError`.

**Начало Части:**
```
[WEBFETCH-ДЕКЛАРАЦИЯ] Начинаю поиск документов по теме «...». Все URL2 будут получены только через реальный web_fetch. Подбор из памяти модели запрещён.
```

**После каждой карточки:**
```
Продолжаю верификацию через web_fetch
```

**Конец Части:**
```
[WEBFETCH-ПОДТВЕРЖДЕНИЕ] Все URL2 в данной Части получены через реальный web_fetch. Заголовки страниц совпадают с наименованиями документов в карточках.
```

**Итоговая схема каждой Части:**
```
[WEBFETCH-ДЕКЛАРАЦИЯ] → web_search → web_fetch → карточка → Продолжаю верификацию через web_fetch → ... → [WEBFETCH-ПОДТВЕРЖДЕНИЕ]
```

## Зафиксированные ошибки и исправления (сессия 04.04.2026)

### Ошибка 1: Неправильное понимание ролей
**Что сказал Qwen:** "notary_agent.py должен сам искать документы, сам генерировать текст, передаю управление агенту"
**Правильно:** notary_agent.py никогда не генерировал текст самостоятельно. Он — оркестратор. Substantive-текст всегда генерировал LLM (человек → Codex → Qwen).
**Исправление:** Зафиксировано в разделе "Правильные роли" выше.

### Ошибка 2: Прямая запись в 02-stage-outputs/
**Что сделал Qwen:** Скопировал part-06-full.md напрямую в 02-stage-outputs/part-06.md через `copy /Y`.
**Правильно:** Прямая запись в 02-stage-outputs/ ЗАПРЕЩЕНА (AGENTS.md). Единственный разрешённый способ — `capture-part-output`.
**Исправление:** Удалил прямой файл, захватил через `capture-part-output`.

### Ошибка 3: Неправильный вывод об отсутствии пакетов для Частей 6–11
**Что сказал Qwen:** "Для Частей 6–11 нет отдельных prepare-команд → пакетов нет вообще → генерирую напрямую."
**Правильно:** Команд `prepare-part-06-plan` и т.д. действительно нет. Но пакеты для Частей 6–11 автоматически пересобираются внутри `capture-part-output` предыдущей части (например, при захвате Части 5 скрипт сам генерирует follow-up пакеты для Частей 6–11). Без отдельной команды.
**Исправление:** Для Частей 6–11 генерировать текст и захватывать через `capture-part-output` — пакеты пересоберутся автоматически.

### Ошибка 4: Неправильный формат карточек для Частей 6–9
**Что сделал Qwen:** Использовал формат `1) Вид документа:`, `2) Полное наименование:` и т.д. с нумерацией `1) 2) 3)...` перед каждым полем.
**Правильно:** Валидатор ищет карточки через regex `(?m)(?=^(?:\d+\.\s+)?Вид документа:)`. Поля должны быть без номера:
```
Вид документа: Федеральный закон
Полное наименование: ...
Орган принятия: ...
Дата и номер: ...
Структурный элемент: ...
Статус: действует
Статус верификации: VERIFIED URL2=ДА
URL1: `http://...`
URL2: `https://...`
Заголовок страницы URL2: ...
Сверка реквизитов: ...
```
**Исправление:** Убрать нумерацию `1) 2) 3)...` перед полями карточки. Оставить только `Вид документа:` в начале строки.

### Ошибка 5: URL1/URL2 в fenced code-блоках не видны валидатору
**Что сделал Qwen:** Писал URL внутри fenced code-блоков:
```
— URL1:
```
http://...
```
```
**Правильно:** Валидатор ищет строки `URL1:` и `URL2:` через regex `^(?:\d+\.\s+)?URL1:\s*(.*)$`. Внутри fenced code-блоков он их не находит. Также нельзя писать `— URL1:` — строка должна начинаться с `URL1:`, без префикса `— `.
**Исправление:** URL1 и URL2 писать в начале строки, без `— `, с URL в inline code:
```
URL1: `http://pravo.gov.ru/proxy/ips/?docbody=&nd=...`
URL2: `https://www.consultant.ru/document/...`
```

### Ошибка 6: FAIL-SAFE CHECK в Части 2 содержал элементы Части 4
**Что сделал Qwen:** В FAIL-SAFE CHECK Части 2 добавил пункты 12–14: "Росфинмониторинг — не выявлено, ФНС — не выявлено, МВД — не выявлено" и т.д.
**Правильно:** FAIL-SAFE CHECK — это короткий блок: что пройдено, что может быть упущено, нужен ли дополнительный поиск. Перечисление органов — это функция Части 4.
**Исправление:** Убраны пункты 12–14 из FAIL-SAFE CHECK (пользователь удалил сам).

### Ошибка 7: Внутренний монолог попал в текст файла
**Что сделал Qwen:** Написал "Now I need to correct the document" и другие внутренние реплики.
**Правильно:** В substantive-файлы попадать должен только канонический текст Части.
**Исправление:** Проверять файлы перед записью. В данном случае артефакт попал только в чат, не в файлы.

## Сессия 04.04.2026 — Системные ошибки при публикации 16.9.1

### Что произошло

Пользователь сказал «публикуй как есть» при недостигнутых метриках. Qwen вместо `--force` начал цепочку самолечения, которая последовательно ухудшала ситуацию:

1. Удалил `manifest.json` → все части стали `unknown`, публикация заблокирована полностью.
2. Запустил regex-фикс по `tmp\`-файлам без бэкапа → файлы Parts 4–10 потеряли все строки URL2.
3. Продолжал чинить следствия собственных ошибок вместо того чтобы остановиться.
4. Задавал вопрос «Продолжаю?» вместо исполнения уже данной инструкции.

Итог: пользователь опубликовал финальный файл вручную сам.

### Правила — никогда не повторять

**Правило 1. `manifest.json` не трогать.**
Это реестр доверия run'а. Удаление переводит все части в статус `unknown` и блокирует публикацию полностью. Никаких исключений.

**Правило 2. Перед patch/regex на source-файлах — сделать копию.**
Сначала `copy file.md file.bak.md`, потом чинить. Без бэкапа — не трогать.

**Правило 3. «Публикуй как есть» = `--force`, без обсуждения.**
Если пользователь принял решение публиковать при недостигнутых метриках — исполнять `--force`. Не предлагать альтернативы, не чинить контент.

**Правило 4. Цепочка самолечения запрещена.**
Если один шаг не прошёл — остановиться, доложить пользователю состояние, ждать команды. Не продолжать исправлять последствия собственных ошибок новыми действиями.
