## Антигаллюцинационный протокол (07.04.2026) — ОБЯЗАТЕЛЕН ДЛЯ КАЖДОЙ ЧАСТИ 2–9

### Уровень 1 — Пороги URL

| Метрика | Порог | Режим |
|---------|-------|-------|
| URL2 | 35 | жёсткий блокер |
| URL2 advisory | 50 | ориентир |
| URL1 | 45 | только ориентир |

Не гнаться за счётчиком — каждая карточка только через реальный поиск.

### Уровень 2 — Commitment Protocol: маркер поиска перед каждой карточкой

Перед каждой карточкой в тексте Части обязан стоять маркер поиска. Допустимые форматы:
```
>>> ПОИСК
[ПОИСК #N]
WEB_SEARCH: ...
Запрос #N:
Продолжаю верификацию через web_fetch
```
Требование: строго **1:1** — один маркер на одну карточку. `search_count < card_count` → `RuntimeError` при захвате.

### Уровень 3 — Grounded Research: один поиск → одна карточка

Цикл работы — без исключений:
```
web_search → web_fetch → ОДНА карточка → web_search → web_fetch → ОДНА карточка → ...
```
Накапливать результаты поиска и писать карточки блоком — запрещено.

### WebFetch Protocol — три обязательных маркера в теле Части

Все три маркера пишутся **внутри текста Части** (не в чате). Валидатор проверяет их при `capture-part-*`. Отсутствие любого → `RuntimeError`.

**Начало Части:**
```
[WEBFETCH-ДЕКЛАРАЦИЯ] Начинаю поиск документов по теме «...». Все URL2 будут получены только через реальный web_fetch. Подбор из памяти модели запрещён.
```

**После каждой карточки:**
```
Продолжаю верификацию через web_fetch
```

**Конец Части:**
```
[WEBFETCH-ПОДТВЕРЖДЕНИЕ] Все URL2 в данной Части получены через реальный web_fetch. Заголовки страниц совпадают с наименованиями документов в карточках.
```

**Итоговая схема каждой Части:**
```
[WEBFETCH-ДЕКЛАРАЦИЯ] → web_search → web_fetch → карточка → Продолжаю верификацию через web_fetch → ... → [WEBFETCH-ПОДТВЕРЖДЕНИЕ]
```

## Сессия 04.04.2026 — Системные ошибки при публикации 16.9.1

### Что произошло

Пользователь сказал «публикуй как есть» при недостигнутых метриках. Qwen вместо `--force` начал цепочку самолечения, которая последовательно ухудшала ситуацию:

2. Запустил regex-фикс по `tmp\`-файлам без бэкапа → файлы Parts 4–10 потеряли все строки URL2.
3. Продолжал чинить следствия собственных ошибок вместо того чтобы остановиться.
4. Задавал вопрос «Продолжаю?» вместо исполнения уже данной инструкции.

Итог: пользователь опубликовал финальный файл вручную сам.

### Правила — никогда не повторять

**Правило 1. `manifest.json` не трогать.**
Это реестр доверия run'а. Удаление переводит все части в статус `unknown` и блокирует публикацию полностью. Никаких исключений.

**Правило 2. Перед patch/regex на source-файлах — сделать копию.**
Сначала `copy file.md file.bak.md`, потом чинить. Без бэкапа — не трогать.

**Правило 3. «Публикуй как есть» = `--force`, без обсуждения.**
Если пользователь принял решение публиковать при недостигнутых метриках — исполнять `--force`. Не предлагать альтернативы, не чинить контент.

**Правило 4. Цепочка самолечения запрещена.**
Если один шаг не прошёл — остановиться, доложить пользователю состояние, ждать команды. Не продолжать исправлять последствия собственных ошибок новыми действиями.

## ПРАВИЛО ОБОСНОВАННОГО ОТСУТСТВИЯ (версия от 12.04.2026) — ОБЯЗАТЕЛЕН ДЛЯ ЧАСТЕЙ 2–10

Любое утверждение в тексте Части о том, что документы **«не выявлены»**, блок **«не применим»**, слой **«пуст»** или новых данных **«не найдено»**, считается **результатом работы**, а не способом её избежать.

Такой вывод разрешён **только** при условии, что в `research-log.jsonl` зафиксированы записи реальных поисковых запросов (`web_search` → `web_fetch`) по соответствующей теме/блоку.

Писать «не выявлено» без предшествующей попытки поиска по релевантным ключевым словам — запрещено. **Отсутствие результата должно быть доказано наличием процесса поиска.**

### Цикл работы: web_search → web_fetch → карточка

1. Перед **каждой** карточкой документа выполни реальный `web_search` → `web_fetch`.
2. **Немедленно** запиши результат в `research-log.jsonl`.
3. Только после успешного `web_fetch` формируй карточку с `VERIFIED URL2: ДА`.
4. Не пиши «не выявлено» пока реально не проверил источники.
5. Честно признавай, когда блок объективно не релевантен, а не пиши «не применимо» чтобы избежать работы.

Этот цикл обязателен для **каждого** документа в Части. Пропуск цикла = обман = блокировка capture.

## Железное правило: каждый URL2 — только через web_fetch перед записью (сессия 12.04.2026)

### Проблема, которую решает правило

12.04.2026 при сборке подтемы `17.8. Порядок нотариального удостоверения сделки двумя и более нотариусами` из 53 проверенных ссылок **21 была санирована** (MISMATCH) — заголовки страниц не совпали с тем, что LLM вписал в карточки. Причина: LLM **экономил шаги** и писал карточки без предварительного `web_fetch` каждой ссылки, угадывая или сокращая заголовок страницы. Результат — бракованный финальный файл.

### Решение: константа `FETCH_EACH_URL2_BEFORE_WRITE` в notary_agent.py

В файл `notary_agent.py` добавлена новая константа `FETCH_EACH_URL2_BEFORE_WRITE` — развёрнутый блок инструкций на русском языке, который автоматически встраивается во **все пакеты LLM** для Частей 2–11.

**Содержание константы:**

- **Алгоритм из 5 шагов для каждой карточки:**
  1. `web_search` → получил URL-кандидат.
  2. `web_fetch` ЭТОГО URL → открыл страницу.
  3. УВИДЕЛ заголовок страницы. СКОПИРОВАЛ его полностью, без сокращений.
  4. Вставил скопированный заголовок в поле «Заголовок страницы URL2» карточки.
  5. ТОЛЬКО ТЕПЕРЬ вписал URL2 в карточку.

- **Явные запреты:**
  - ❌ Угадывать заголовок страницы по названию документа.
  - ❌ Сокращать заголовок страницы.
  - ❌ Писать URL2 в карточку БЕЗ предшествующего `web_fetch` этой ссылки.
  - ❌ Писать карточки пачкой, а `web_fetch` делать потом или вообще не делать.
  - ❌ Копировать URL2 из другой карточки или из предыдущей подтемы.
  - ❌ Писать пустой «Заголовок страницы URL2» или оставлять поле пустым.

- **Правильный цикл (один документ):**
  `web_search → web_fetch → увидел заголовок → записал карточку → web_search → web_fetch → увидел заголовок → записал карточку → ...`

- **Неправильный цикл (блок):**
  `web_search × 10 → написал 10 карточек → потом web_fetch (или забыл).`

- **Правило длинных заголовков:** если заголовок страницы содержит название сайта (например `... \ КонсультантПлюс`) — пиши ВЕСЬ заголовок, это НЕ MISMATCH.
