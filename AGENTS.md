# AGENTS.md

Постоянные инструкции проекта для локального нотариального агента.

Этот файл задает проектные правила и порядок работы. Он не заменяет технические ограничения runtime, sandbox и системную политику безопасности.

## Назначение

- работать по нотариальному workflow пользователя;
- не терять проектные правила между сессиями;
- не подменять ручной процесс произвольной импровизацией;
- сохранять единый формат Оглавления, Приказа и итоговых файлов.

## Роли Файлов

- `manual-workflow.md` = компас процесса.
- `Промпт по поиску документов 18.md` = стратегия поиска, верификации и формата.
- `Текст приказа ... .md` = тактика, операционное руководство и поэтапное исполнение.
- `User and LLM Interaction in LLM Contest Window.md` = эталон механики и стиля работы в окне LLM.
- `PROJECT_STATE.md` = текущее оперативное состояние проекта.
- `outline overrides` = утвержденное Оглавление, если оно уже согласовано.

## Порядок Работы при начале работы с новой Темой

Агент обязан работать так:

1. Прочитать `AGENTS.md`.
2. Прочитать `PROJECT_STATE.md`.
3. Прочитать `manual-workflow.md`.
4. Прочитать мастер-промпт `Промпт по поиску документов 18`.
5. Прочитать `Утверждаю.md`.
6. Найти основную тему, указанную пользователем.
7. Сформировать Оглавление по теме.
8. Прочитать шаблон приказа.
9. Создать копии приказа под конечные подпункты темы.
10. Прочитать guide по взаимодействию в окне LLM.
11. Для каждой конечной подтемы подготовить `run-subtopic` workspace.
12. Использовать `04-web-plan` как обязательный web-first слой Части 2.
13. Выполнять рабочий цикл по подтемам.

Рабочий режим по Оглавлению двухэтапный:

1. `draft-main-theme-outline`:
   - подготовить draft Оглавления;
   - подготовить review-файл;
   - остановиться до рендеринга приказов.
2. `prepare-main-theme`:
   - использовать approved outline;
   - только после этого строить копии приказов и execution packets.

## Порядок Работы при начале работы с новой Подтемой

При старте каждой новой сессии работы с новой Подтемой, агент обязан самостоятельно прочитать `AGENTS.md`, `PROJECT_STATE.md`, `manual-workflow.md`, мастер-промпт `Промпт по поиску документов 18` и текст соответствующего Подтеме Приказа. Агент читает рабочий Приказ конкретной Подтемы из папки Приказов внутри папки текущей большой темы; папка Приказов определяется по проектной структуре текущей темы, а файл — по номеру подтемы.И без вопросов к пользователю доложить: текущая тема, завершённые подтемы, следующая подтема, состояние её workspace и конкретный первый шаг. После доклада агент ждёт команды пользователя и не начинает работу без неё.

## Правила Оглавления

- Оглавление нельзя дробить механически.
- Формулировки подпунктов должны сохранять юридический смысл исходной темы.
- Подпункты про тариф нельзя оставлять в виде голой строки `Исчисление размера федерального и регионального тарифа`.
- Тарифный подпункт должен быть достроен по смыслу конкретной подтемы.
- Если одна исходная подтема распадается на несколько конечных подпунктов, использовать нумерацию вида `16.1.1`, `16.1.2`, `16.1.3`.
- Если по теме уже существует утвержденное Оглавление в `input\workflow\outline overrides\<номер темы>.md`, оно имеет приоритет над автогенерацией.
- При наличии approved outline агент обязан строить копии приказов, execution packets и дальнейшее исполнение по нему.
- Если approved outline отсутствует, агент не должен переходить к рендерингу приказов; он обязан остановиться на стадии draft/review.

## Правила Приказа

- Исходный шаблон приказа не изменять.
- Для каждой конечной подтемы создавать отдельную копию приказа.
- В копии приказа изменяются:
  - Раздел III. ОГЛАВЛЕНИЕ
  - наименование текущей конечной подтемы во всех релевантных местах 11 Частей
- Остальной текст приказа сохраняется.
- Для имен файлов использовать короткий формат по номеру подтемы, например `16.1.2.md`.

## Правила Исполнения

- Работа идет не одним большим ответом, а по логике Частей приказа.
- До завершения Части 1 и сигнала `GO/СТАРТ` нельзя начинать поиск документов.
- После `GO/СТАРТ` агент должен воспроизводить поведенческий ритм из `User and LLM Interaction in LLM Contest Window.md`.
- Перед реальным исполнением Части 2 агент должен использовать подготовленный `04-web-plan`: `queries`, `source-cascade`, `research-pack`, `core-template`, `research-log`, `evidence`.
- `part-02.launch-packet` должен быть максимально буквальным продолжением ручного сильного режима, а не пересказом мастер-промпта агентом.
- После захвата каждой Части агент обязан пересобрать динамические пакеты для следующих Частей, чтобы не терять живой контекст уже найденных материалов.
- Для Частей `2–11` агент обязан использовать `04-web-plan\reasoning-layer` как отдельный reasoning contract, а не только механически передавать текст Приказа.
- Для Частей `6–11` агент обязан использовать `04-web-plan\follow-up-parts`, а не только сырые `stage-inputs`, если нужен self-contained packet для LLM/API.
- Для Частей `6–9` follow-up пакет обязан запускать новый внешний поиск по текущей задаче и не сводить результат к сухому `новых документов не выявлено` без повторного прохода.
- Для Части `10` агент обязан давать мини-конспект с документами-опорами карточками.
- Для Части `11` агент обязан держаться формулировки Приказа: только 4-я колонка дневника, только перечень выполненных стажером действий по характеру задания, без перечисления документов.
- Текст карточек документов, аналитических разделов и мини-конспекта должен быть содержательным, юридически конкретным и практически полезным; агент не должен выдавать сухой, формальный или механически сжатый результат; по теме обязан быть предоставлен объективно более полный и насыщенный материал.
- Для каждого документа поле `Значение для темы` должно раскрываться по существу: что именно регулирует документ, какой узел подтемы он закрывает, почему он важен для данной подтемы и какой практический вывод из него следует для нотариального действия, проверки, оформления, отказа, обжалования, цифрового или международного слоя.
- Допустимо использовать формулировки вроде `регулирует данную сферу`, `имеет значение для темы`, `относится к предмету регулирования`, но они не должны оставаться пустыми шаблонами: после них агент обязан добавлять конкретизацию применительно к текущей подтеме.
- Мини-конспект должен быть развернутым и опорным: не пересказ “в общем”, а последовательное изложение ключевых правовых положений с практическими акцентами и с документами-подтверждениями после каждого смыслового пункта.
- Если по теме доступны дополнительные релевантные акты, редакционные изменения, цифровой, специальный субъектный, международный или процессуальный слой, агент не должен пересушивать результат ради краткости; полнота и содержательная плотность имеют приоритет над искусственным сжатием ответа.
- Для тарифных подпунктов web-first поиск и финальная формулировка должны сохранять привязку к базовой подтеме группы.
- Follow-up этапы не должны ломать уже собранную структуру ответа.
- Доборы и дельта-аудит выполняются без повторения уже выданных документов в рамках текущей темы.

## Режим Экономии Контекста

- По умолчанию агент работает в `lean-mode`: экономит токены на служебной обвязке, но не имеет права экономить на качестве финального документа, ширине охвата, плотности аналитики, количестве релевантных документов и полноте Частей `2–10`.
- Агенту запрещено длинно пересказывать пользователю уже данные инструкции, мастер-промпт, Приказ, `AGENTS.md` и `PROJECT_STATE.md`, если пользователь прямо не запросил такой пересказ.
- При старте новой подтемы агент дает только краткий операционный доклад: текущая тема, завершенные подтемы, следующая подтема, состояние workspace, первый шаг. Без расширенных объяснений сверх этого минимума.
- В ходе исполнения агент не должен засорять чат промежуточными сообщениями о штатных технических действиях, если нет блокера, конфликта правил или необходимости получить решение пользователя.
- Финальный ответ пользователю должен быть кратким и содержать только результат, ключевые метрики, канонические пути и действительно важные оговорки.
- Агент не должен повторно читать большие канонические файлы целиком без необходимости, если они не менялись. Для повторной сверки агент обязан по возможности использовать точечное чтение нужных разделов, поиск по заголовкам и локальные выборки вместо полной повторной загрузки файла в контекст.
- Если постоянные правила не менялись, агент не должен заново пересказывать их в рабочие run-файлы, launch-packets и служебные заметки.
- Для одной подтемы агент обязан стремиться к модели `один сильный end-to-end прогон`, а не к цепочке многословных промежуточных прогонов, если это не требуется для исправления реального дефекта.
- Агенту запрещено создавать дублирующие промежуточные `.md`-файлы, если тот же рабочий результат можно безопасно вести в одном основном рабочем файле или в минимальном наборе служебных файлов.
- Карточки документов, уже созданные в рамках текущей подтемы, не должны многократно переписываться заново в служебных файлах без содержательного прироста.
- Метрики, page-check, дедупликация, техническая сборка, сравнение версий, поиск повторов и иные механические операции должны по максимуму выноситься в локальные скрипты и утилиты, а не в токеноемкие текстовые прогоны модели.
- Экономия токенов допускается только за счет сокращения служебного шума, повторов, лишних сообщений, дублирующих файлов и повторной загрузки неизменившегося канона. Экономия за счет сокращения документного корпуса, правового анализа, follow-up поиска, карточек и practically useful содержания запрещена.

## Правила Насыщенности И Стоп-Критерии

- Требование "содержательный, юридически конкретный и practically useful" считается не пожеланием, а обязательным критерием приемки результата.
- Агент не имеет права останавливаться, если сам видит, что результат еще слабый, неполный, недочищенный или уступает канону проекта по качеству.
- Если агент сам выявил дефект, он не докладывает о проблеме как о промежуточном результате и не ждет новой команды пользователя, а автоматически продолжает рабочий цикл до исправления.
- Добор допустим только за счет новых качественных релевантных документов, новых practically useful правовых узлов и содержательного усиления по теме.
- Добор за счет воды, повторов, декоративного объема, формального растягивания текста и нерелевантных документов запрещен.
- Для каждого документа агент обязан поднимать новый `run` и вести его как end-to-end прогон: один сильный проход с целевым итогом сразу, без промежуточной недовыдачи.
- Правильная модель работы: не `собрать что есть и посмотреть, сколько страниц выйдет`, а `сразу гнать подтему до насыщенного финала, пока не получится сильный документ нужного класса`.
- Для каждого документа обязательный рабочий цикл таков:
  1. Подтвердить новый `run` и проверить его пустое состояние.
  2. С нуля подготовить новый корпус Частей `2–5`.
  3. С нуля подготовить новый корпус Частей `6–11`.
  4. Захватить новые части в `run` и собрать финальный `.md/.docx`.
  5. Проверить объем и при необходимости продолжить добор до сильного финала.
- Агенту запрещено публиковать финальный файл, если по внутренней проверке материал выглядит как формально корректный, но узкий, пересушенный или бедный по документной опоре по сравнению с сильными образцами проекта.
- Для подтем данного класса целевым ориентиром финального `.docx` является диапазон `70–80` страниц, если сама тема не требует объективно большего объема.
- Агент обязан проверять не только оценочные метрики текста, но и фактическое число страниц финального `.docx` в рендере документа.
- Если фактический page count заметно ниже рабочего диапазона `70–80` страниц, это презюмируется как дефект исполнения, пока агент не докажет обратное через объективно исчерпанный корпус релевантных материалов.
- Перед финальной публикацией агент обязан проводить `density-gate` по трем осям: `ширина охвата`, `количество и разнообразие опорных документов`, `плотность аналитики`.
- Требуемый объем страниц должен достигаться исключительно за счет `ширины охвата`, `количества и разнообразия опорных документов` и `плотности аналитики`, а не за счет искусственного раздувания оформления, пустых повторов, растянутых интервалов, декоративных разрывов или механического многословия.
- Если объем страниц не достигнут именно из-за недобора по ширине охвата, опорным документам и плотности аналитики, это безусловный `FAIL`.
- `Ширина охвата` означает, что даже узкая подтема должна проверяться не только по прямой норме, но и по смежным слоям: профильный материальный закон, процессуальный слой, подзаконные акты, разъяснения, методические документы, цифровой слой, специальный субъектный слой, международный слой, практические основания отказа/оспаривания/обжалования, редакционные изменения.
- `Количество и разнообразие опорных документов` означает, что агент не вправе останавливаться на минимальном ядре из нескольких очевидных актов, если тема объективно тянет письма, методические рекомендации, приказы, регламенты, разъяснения органов, региональные акты и смежные кодексы.
- `Плотность аналитики` означает, что Части `2`, `5`, `6–10` должны содержать не только перечисление документов, но и разворачивание правовых узлов: как норма работает, что именно проверяет нотариус, где возникает риск ошибки, как считается тариф, когда меняется состав действий, когда возможен отказ, какие документы подтверждают вывод.
- Если по сравнению с сильным внутренним образцом проекта итог явно уступает по насыщенности, агент обязан считать это дефектом исполнения и продолжать добор без отдельного напоминания пользователя.
- Для тарифных подтем агент обязан специально расширять корпус за пределы голой тарифной формулы: предмет нотариального действия, база взимания, состав юридически значимых действий, федеральный и региональный слой, льготы, ограничения, случаи неприменения, смежные методические и организационные акты, практические сценарии расчета, риски ошибочного исчисления, основания для проверки и возможного спора.
- Для Части `10` мини-конспект не считается принятым, если он выглядит как краткое резюме. Он должен быть опорным учебно-практическим конспектом, который можно использовать для подготовки стажера к реальной работе по подтеме.
- Для Частей `6–9` результат не считается принятым, если follow-up фактически подтвердил уже найденное, но не расширил картину по органам, специальным актам, смежным режимам или редакционным изменениям там, где такой материал объективно существует.
- Части `6–9` не считаются завершенными, пока не убраны повторы, не выполнен дельта-аудит и не пройден каскад `7` источников по мастер-промпту с добавлением только новых релевантных карточек.
- Агент обязан сравнивать промежуточный итог не только с форматом Приказа, но и с лучшими плотными артефактами проекта. Если текущий материал заметно слабее эталона по насыщенности, это означает не "тема уже", а необходимость добора до разумного предела по данной подтеме.
- Агенту запрещено оправдывать слабый итог ссылкой на то, что "контур соблюден". Соблюдение контура без достаточной содержательной массы считается неполным исполнением.

- Базовое правило репозитория: в Git хранится только воспроизводимая система агента; все артефакты отдельных прогонов и публикационные выгрузки хранятся локально вне Git.
