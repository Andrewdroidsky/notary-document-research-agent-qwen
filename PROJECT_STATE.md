## Новые Правила Финального Формата

- Начиная со следующей подтемы, финальная сборка обязана выводить явные заголовки `ЧАСТЬ 2` ... `ЧАСТЬ 11`.
- В карточечных разделах документы должны нумероваться.
- В итоговом `.md` ссылки публикуются в эталонном читабельном виде: `URL1`, `URL2`, `VERIFIED URL2`, `Заголовок страницы URL2`, `Сверка реквизитов`, без обязательных fenced code blocks.
- В `.docx` эти поля выводятся как структурированные моноширинные блоки без markdown-маркеров fenced code blocks.
- Любой названный документ обязан иметь карточку с полным наименованием, структурным элементом и ссылочным блоком.
- Поиск по органам власти публикует карточки документов, а не групповые строки `Прямой документ(ы): ...`.
- Канонические финальные файлы публикуются по схеме `номер подтемы + краткий заголовок из Оглавления`.
- Краткий заголовок для имени файла должен браться из утвержденной формулировки конечной подтемы, а не из произвольного сокращения агентом.
- Модель верификации изменена: слабый `URL2` больше не выбрасывает реально найденный документ, если уже подтверждены существование, применимость, роль и `URL1`; в таком случае снижается статус доверия карточки, а не полнота корпуса.

Пользователь может при необходимости просить обновить этот файл, если меняются правила, пути или формат результата.

## Критическое Правило Проверки Состояния (с 04.04.2026)

- Перед **любым** действием по подтеме агент обязан сначала проверить **фактическое состояние** `02-stage-outputs/`.
- Агент **не имеет права** полагаться на память, предположения или механический порядок частей.
- В код `notary_agent.py` встроены функции:
  - `get_stage_outputs_status(run_workspace)` — проверяет наличие и содержательность каждой из 11 частей, возвращает `dict[int, str]` со статусами `present` / `absent`.
  - `report_stage_outputs_status(status)` — выводит отчёт: какие части захвачены, какие отсутствуют.
  - `require_stage_part_not_present(run_workspace, part_number, command)` — блокирует попытку захвата уже готовой части.
  - `require_stage_part_present(run_workspace, part_number, command)` — блокирует работу с ещё не захваченной частью.
- Проверка встроена во все команды: `execute-part-01`, `capture-part-output`, `prepare-part-02-web`, `prepare-part-03-plan`, `prepare-part-04-plan`, `prepare-part-05-plan`, `capture-part-03-range`, `capture-part-04-range`, `capture-part-05-range`, `assemble-subtopic-final`.
- Если часть уже захвачена — агент **обязан** перейти к следующей отсутствующей части, а не повторять работу.
- Нарушение этого правила считается дефектом orchestration и приводит к потере токенов и путанице в отчётах.
